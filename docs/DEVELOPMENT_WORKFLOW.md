# Development Workflow

> Версия: 1.0
> Статус: Stable

## Назначение

Данный документ определяет единый процесс разработки Platform Core
и всех пользовательских каналов. Цель — обеспечить одинаковое
качество изменений независимо от того, кто их выполняет:
разработчик, внешний подрядчик или AI-ассистент.

## Основной принцип

Любое изменение проходит один и тот же жизненный цикл. Изменение
считается завершённым только после прохождения всех этапов
проверки.

## Общий процесс
Изучение задачи

↓

Анализ существующей архитектуры

↓

Подготовка решения

↓

Реализация

↓

Автономное тестирование

↓

Ручная проверка

↓

Формирование отчёта

↓

Интеграция

↓

Pull Request

Интеграция не должна выполняться до завершения автономной
проверки.

## Этап 1. Анализ

Определить: какой пакет изменяется; относится ли изменение к
Platform Core; можно ли использовать уже существующий компонент;
не нарушаются ли архитектурные правила. При необходимости изучить
ARCHITECTURE.md, PLATFORM_RULES.md, REPOSITORY_MAP.md.

## Этап 2. Реализация

Изменения должны быть минимальными. Каждый Pull Request должен
иметь одну ответственность. Не допускается смешивание рефакторинга,
новой функциональности и исправления ошибок.

## Этап 3. Автономное тестирование

До подключения прикладного приложения изменение проверяется через
платформу: Emulator, Scenario Engine, Execution Log, Verification
Harness.

## Этап 4. Validation Bench

После успешной автоматической проверки выполняется ручная проверка:
пользовательский интерфейс, распознавание речи, синтез речи,
выполнение сценариев, журнал выполнения.

## Этап 5. Проверка сценариев

Validation Bench поддерживает режимы: Автоматический (все шаги
последовательно), Пошаговый (каждый шаг вручную), Повтор шага,
Переход к следующему шагу.

## Этап 6. Формирование отчёта

После проверки формируется JSON Validation Report: информация о
тестировщике, параметры окружения, информация о сборке, результаты
Verification, Execution Log, продолжительность проверки.

## Этап 7. Отправка отчёта

Через штатный Backend API: авторизация через `/api/v1/auth/`,
получение token и u_hash, формирование JSON Report, вызов
`/api/v1/mail/`, отправка JSON как вложения. Validation Bench не
реализует собственную авторизацию и почтовую систему.

## Этап 8. Интеграция

Только после успешного прохождения всех предыдущих этапов изменение
интегрируется в прикладную систему (Taxi, Delivery, Marketplace).

## Этап 9. Pull Request

Каждый PR должен содержать: описание задачи, перечень изменений,
результаты сборки, результаты проверки, ограничения, технический
долг (если имеется).

## Требования к Validation Bench

Выбор языка; ввод имени тестировщика; выбор набора сценариев; выбор
режима выполнения; отображение текущего шага и прогресса; повтор
шага; переход к следующему шагу; просмотр журнала выполнения и
результатов Verification; формирование JSON Report; отправка отчёта
через Backend API. При необходимости стенд разворачивается на
сервере и доступен по ссылке без локальной установки.

## Критерий завершения работы

Изменение считается завершённым только тогда, когда: проект успешно
собирается; автономные сценарии проходят без ошибок; Validation
Bench подтверждает корректность работы; сформирован Validation
Report; отчёт успешно отправлен через Backend API; изменение принято
в основную ветку проекта.
